# Explanation Schema

This document defines the schema used by AIthera’s Explanation Generator to describe how and why a final educational prompt was created. It provides the contract for auditability, transparency, and backend persistence.

---

## 1. Purpose

The Explanation Schema captures the reasoning behind the final prompt generated by the Consensus Builder. It documents:

- which AI models contributed and how,
- why their contributions were selected,
- how the learning style influenced the final output,
- the consensus decision rationale.

This schema enables transparent API responses, structured audit logs, and rerunnable explanation workflows.

---

## 4. Field Definitions

### summary

- Type: `string`
- Required: `Yes`
- Description: A concise overview of why the final prompt was created and what it emphasizes.

### decision_rationale

- Type: `string`
- Required: `Yes`
- Description: A narrative explanation of the consensus decision, including selection criteria and trade-offs.

### contributions

- Type: `array`
- Required: `Yes`
- Description: Attribution records for each AI provider’s contribution to the final prompt.
- Each item includes:
  - `provider`: AI model identifier.
  - `response_id`: original response identifier.
  - `contribution_type`: role of the contribution such as `structure`, `reasoning`, `example`, or `precision`.
  - `selected`: whether the contribution was included in the final prompt.
  - `why_selected`: rationale for selection.
  - `excerpt`: optional snippet representing the contribution.

### learning_style_adaptation

- Type: `object`
- Required: `Yes`
- Description: Explains how the final prompt was adjusted for the learner’s style.
- Fields:
  - `style`: learning style identifier.
  - `adaptation_summary`: concise description of applied adaptations.
  - `adaptation_details`: optional list of specific adaptation actions.

### consensus_trace

- Type: `object`
- Required: `Yes`
- Description: Trace data for how the consensus was formed.
- Fields:
  - `selected_sources`: providers or response identifiers selected.
  - `conflicts_resolved`: explanation of any conflicting recommendations.
  - `resolution_strategy`: rules or heuristic used to resolve conflicts.

### metadata

- Type: `object`
- Required: `Yes`
- Description: Processing metadata for auditing and lifecycle management.
- Fields:
  - `created_at`: ISO 8601 timestamp.
  - `version`: schema or explanation generator version.
  - `processing_time_ms`: time taken to generate the explanation.

## 5. Required Fields

- `summary`
- `decision_rationale`
- `contributions`
- `learning_style_adaptation`
- `consensus_trace`
- `metadata`

Nested required fields:

- `contributions[].provider`
- `contributions[].response_id`
- `contributions[].contribution_type`
- `contributions[].selected`
- `contributions[].why_selected`
- `learning_style_adaptation.style`

- `consensus_trace.selected_sources`
- `consensus_trace.resolution_strategy`
- `metadata.created_at`
- `metadata.version`
- `metadata.processing_time_ms`

## 6. Optional Fields

- `contributions[].excerpt`
- `learning_style_adaptation.adaptation_details`
- `consensus_trace.conflicts_resolved`

## 7. Validation Rules

The Explanation Generator must validate outputs before persistence.

| Rule | Description | Failure Response |
|---|---|---|
| Required field presence | All required fields must exist | `{"error":"Missing required field","field":"summary"}` |
| String type validation | Strings must be valid text | `{"error":"Invalid field type","field":"decision_rationale","expected":"string"}` |
| Boolean type validation | `contributions.selected` must be boolean | `{"error":"Invalid field type","field":"contributions[0].selected","expected":"boolean"}` |
| Date-time format | `metadata.created_at` must be ISO 8601 | `{"error":"Invalid date format","field":"metadata.created_at"}` |
| Score range not applicable | Not applicable for explanation schema | N/A |

### Additional validation behaviors

- Trim whitespace in string fields.
- Normalize arrays and remove empty entries from `adaptation_details`.
- Preserve explanation text verbatim after sanitization for safe storage.

## 8. AI Contribution Attribution

The schema supports explicit attribution of provider contributions.

### Attribution guidelines

- `provider` identifies the model: `openai_gpt`, `anthropic_claude`, `google_gemini`, `deepseek`.
- `response_id` links back to the original provider response.
- `contribution_type` categorizes the contribution by role.
- `selected` indicates whether the contribution was used.
- `why_selected` documents the rationale.
- `excerpt` captures a representative snippet.

### Attribution rules

- Always include a record for each provider that contributed to consensus decisions.
- Include multiple records when a provider contributes more than one role.
- Mark contributions not selected when they were considered but omitted.
- Use `why_selected` to record both positive and negative selection rationale.

---

## 9. Learning Style Adaptation Tracking

This section documents how the final prompt was adjusted for the specific learner style.

### Tracking elements

- `style`: the active learning style.
- `adaptation_summary`: why the prompt was shaped for that style.
- `adaptation_details`: concrete adaptation actions.

### Example adaptation patterns

- Visual: "Used headings, bullets, and diagram metaphors."
- Conversational: "Added questions and interactive tone."
- Step-by-step: "Converted content into ordered procedural steps."
- Exam-focused: "Focused on concise review items and practice questions."
- Research-oriented: "Added comparative analysis and discussion prompts."

## 11. Database Storage Recommendations

Use the Explanation Schema as the storage contract for explanation artifacts.

### Recommended storage model

- `summary`, `decision_rationale`: store as text.
- `contributions`: store as JSON array or normalized attribution table.
- `learning_style_adaptation`: store as JSON.
- `consensus_trace`: store as JSON.
- `metadata`: store `created_at`, `version`, and `processing_time_ms` as discrete fields, with `debug_info` if added as JSON.

### Persistence guidelines

- Use a relational table with JSON columns for `contributions`, `learning_style_adaptation`, and `consensus_trace`.
- Index `metadata.created_at` and `metadata.version`.
- Retain explanation artifacts for auditability and performance analysis.
- Keep `why_selected` and `resolution_strategy` in audit logs for traceability.

---

## 12. Future Extensibility

The schema can be extended without breaking existing consumers by adding optional fields.

### Extension guidelines

- Add new optional explanation attributes rather than making current fields required.
- Expand `contributions` records with additional metadata fields such as `contribution_confidence` or `source_role`.
- Add new tracking fields to `learning_style_adaptation` such as `adaptation_reasoning`.
- Extend `consensus_trace` with `reviewed_by` or `review_timestamp`.
- Version the schema when introducing incompatible changes.

This schema supports backend implementation, auditability, and future evolution of the Explanation Generator.
