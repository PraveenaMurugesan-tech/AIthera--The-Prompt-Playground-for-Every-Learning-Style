# Consensus Schema

This document defines the final output contract generated by the Consensus Builder in AIthera. It specifies how multiple AI provider responses are merged into a consensus-backed educational prompt, including source attribution, scoring, learning style tracking, metadata, and audit details.

---

## 1. Purpose

The Consensus Schema provides a stable API and data model for storing the result of the AI Council’s merge process. It enables:

- consistent backend payloads for final prompt delivery,
- auditability of model contributions,
- tracking of learning style adaptation,
- prompt scoring and metadata storage,
- storage and retrieval of source attribution rules.

This schema is intended for backend implementation, API response design, and database modeling.

---

## 2. Consensus Output Structure

The consensus output is a structured object containing the final prompt, contribution metadata from each provider, learning style context, scoring data, and audit information.

```json
{
  "final_prompt": "string",
  "learning_style": "string",
  "score": {
    "clarity": "number",
    "structure": "number",
    "personalization": "number",

    "depth": "number",

  },
  "sources": [
    {
      "provider": "string",
      "response_id": "string",
      "contribution_type": "string",
      "confidence_score": "number",
      "excerpt": "string"
    }
  ],
  "metadata": {
    "created_at": "string",

  "audit": {

```

## 4. Field Definitions

### final_prompt

- Type: `string`
- Required: `Yes`
- Description: The final educational prompt assembled by the Consensus Builder.
- Notes: This is the prompt to deliver to the learner or the next processing stage.

### learning_style

- Type: `string`
- Required: `Yes`
- Description: The learning style context used when generating the final prompt.
- Notes: Should match one of the input learning styles such as `visual`, `conversational`, `step_by_step`, `exam_focused`, or `research_oriented`.

### score

- Type: `object`
- Required: `Yes`
- Description: Normalized scoring values for the final prompt.
- Fields:
  - `clarity`: measures readability and understandability.
  - `structure`: measures logical flow and organization.
  - `personalization`: measures learner-specific adaptation.
  - `effectiveness`: measures pedagogical usefulness.
  - `depth`: measures informational richness.
  - `composite`: aggregated score used for ranking.

### sources

- Type: `array`
- Required: `Yes`
- Description: Attribution records for contributing AI provider outputs.
- Items:
  - `provider`: provider identifier.
  - `response_id`: original response identifier.
  - `contribution_type`: role of the provider’s contribution, e.g., `structure`, `reasoning`, `example`, `precision`.
  - `confidence_score`: provider confidence or contribution weight.
  - `excerpt`: optional text snippet representing the provider’s contribution.

### metadata

- Type: `object`
- Required: `Yes`
- Description: Processing metadata for persistence and diagnostics.
- Fields:
  - `created_at`: timestamp when the consensus output was generated.
  - `consensus_version`: version of the consensus logic or schema.
  - `processing_time_ms`: total time in milliseconds for consensus generation.
  - `debug_info`: optional provider-specific or pipeline-specific debug data.

### audit

- Type: `object`
- Required: `Yes`
- Description: Audit information supporting source attribution and decision rationale.
- Fields:
  - `source_attribution_rules`: description of how source contributions were selected.
  - `resolution_notes`: notes explaining conflict resolution or special processing decisions.

## 5. Required Fields

- `final_prompt`
- `learning_style`
- `score`
- `sources`
- `metadata`
- `audit`

Within nested objects, the following are also required:

- `score.clarity`
- `score.structure`
- `score.personalization`
- `score.effectiveness`
- `score.depth`
- `score.composite`
- `metadata.created_at`
- `metadata.consensus_version`
- `metadata.processing_time_ms`
- `audit.source_attribution_rules`
- `sources[].provider`
- `sources[].response_id`
- `sources[].contribution_type`

- `sources[].confidence_score`
- `sources[].excerpt`
- `metadata.debug_info`
- `audit.resolution_notes`

## 7. Validation Rules

The Consensus Builder must validate output against these rules before persisting or returning it.

| Rule | Description | Failure Behavior |
|---|---|---|
| Required field presence | All required fields must exist | Reject consensus output and return validation error |
| String type validation | All string fields must be strings | Reject invalid field with type error |
| Score range | Score values must be between 0 and 1 | Reject invalid score value |
| Non-empty final prompt | `final_prompt` must not be empty or whitespace | Reject with `final_prompt` invalid error |
| Learning style validity | `learning_style` should match known styles | Reject with supported style list |
| Source attribution completeness | Each source entry must include provider, response_id, and contribution_type | Reject incomplete source attribution record |
| Timestamp format | `metadata.created_at` must be ISO 8601 date-time | Reject invalid date format |

### Sanitization

- Trim whitespace in string fields.
- Normalize `final_prompt` for safe storage.
- Preserve audit and metadata fields in raw form where possible.

## 8. Source Attribution Rules

The Consensus Builder must record how each provider contributed to the final prompt.

### Attribution guidelines

- `provider` identifies the source model.
- `response_id` links back to the provider response.
- `contribution_type` categorizes the contribution:
  - `structure` for layout and flow,
  - `reasoning` for conceptual clarity,
  - `example` for illustrative material,
  - `precision` for technical detail.
- `excerpt` should contain a short representative snippet of the contribution.

### Attribution decision rules

- If GPT provided the outline, record `structure` for GPT.
- If Claude provided conceptual framing, record `reasoning` for Claude.
- If Gemini provided analogies, record `example` for Gemini.
- If DeepSeek provided ordered steps, record `precision` for DeepSeek.
- If a provider contributed to multiple roles, include multiple source records.

## 9. Metadata Fields

### created_at

- Type: `string`
- Format: `date-time`
- Description: ISO 8601 timestamp of consensus generation.

### consensus_version

- Type: `string`
- Description: Version identifier for the consensus logic or schema.
- Example: `"v1.0"`

### processing_time_ms

- Type: `integer`
- Description: Total time spent generating consensus output.
- Notes: Includes AI response processing, analysis, and merge time.

### debug_info

- Type: `object`
- Description: Optional debug payload containing provider or pipeline diagnostics.
- Notes: Useful for incident investigation.

## 11. Error Handling

The Consensus Builder must return structured validation errors for invalid outputs.

### Missing required fields

```json
{
  "error": "Missing required field",
  "field": "final_prompt"
}
```

### Invalid score range

```json
{
  "error": "Invalid score value",
  "field": "score.clarity",
  "minimum": 0,
  "maximum": 1
}
```

### Invalid source attribution

```json
{
  "error": "Invalid source record",
  "field": "sources[0].provider"
}
```

### Invalid timestamp format

```json
{
  "error": "Invalid date format",
  "field": "metadata.created_at"
}
```

## 12. Database Storage Recommendations

Use the Consensus Schema as the primary storage contract for the final prompt artifacts.

### Recommended storage model

- `final_prompt`: store as text.
- `learning_style`: store as string or enum.
- `score`: store as JSON or normalized numeric fields.
- `sources`: store as a JSON array or related attribution table.
- `metadata`: store as JSON with structured fields for timestamps and versioning.
- `audit`: store as JSON for audit trail and reasoning.

### Persistence guidelines

- Use a relational table for core fields and JSON columns for metadata/debug_info.
- Keep `sources` as queryable JSON to allow attribution reporting.
- Index `learning_style`, `metadata.created_at`, and `metadata.consensus_version`.
- Retain `audit` information for compliance and review.

---

## 13. Future Extensibility

The schema should be extended in a backward-compatible way.

### Extension guidelines

- Add new optional fields rather than changing required fields.
- Preserve the existing `final_prompt` and `sources` contract.
- Introduce new score dimensions as optional fields (e.g. `engagement`, `accuracy`).
- Add new audit or metadata properties without altering semantics.
- Version the schema for incompatible changes.

### Example future extensions

- `score.engagement`
- `sources[].contribution_detail`
- `metadata.pipeline_stage`
- `audit.reviewed_by`

This schema is designed for long-term backend implementation, allowing the Consensus Builder to produce auditable, versioned, and analytically rich final prompt artifacts.
